## **ERROR HANDLING STANDARD — รูปแบบจัดการ Error**

### API Response Format
```typescript
{
  success: boolean,
  data: T | null,
  error: {
    code: string,      // "AUTH_001", "DB_001", "VALIDATION_001"
    message: string    // user-friendly message
  } | null,
  meta?: {
    timestamp: string,
    requestId: string
  }
}
```

### Error Handling Rules
* ทุก function ที่อาจ fail → ต้อง try-catch หรือ return Result type
* แยก **user-facing message** (ภาษาที่ user เข้าใจ) กับ **internal error** (technical details)
* Log error พร้อม context: `userId`, `action`, `timestamp`, `stack trace`
* ใช้ Error codes เป็นหมวดหมู่: AUTH_xxx, DB_xxx, VALIDATION_xxx, EXTERNAL_xxx

### Error Logging Format
```typescript
logger.error({
  code: "DB_001",
  message: "Failed to insert user",
  userId: "xxx",
  action: "createUser",
  timestamp: new Date().toISOString(),
  stack: error.stack,
  input: sanitizedInput  // ห้าม log sensitive data
});
```

## **COMPLIANCE & AUDIT — มาตรฐานการตรวจสอบ**

### Audit Logging
```typescript
// ทุก action สำคัญต้อง log เพื่อ audit
auditLog({
  action: "USER_DELETE",
  actor: { id: "admin-123", role: "admin" },
  target: { type: "user", id: "user-456" },
  timestamp: new Date().toISOString(),
  ip: request.ip,
  userAgent: request.headers["user-agent"],
  result: "success",
  metadata: { reason: "requested by user" }
```

### Audit Log Rules
* **ต้อง log:** authentication, authorization, data changes, admin actions
* **ห้าม log:** passwords, tokens, credit card numbers, PII ที่ไม่จำเป็น
* **Retention:** เก็บ audit logs อย่างน้อย 90 วัน (หรือตาม compliance requirement)
* **Immutable:** audit logs ห้ามแก้ไข/ลบ

### Data Classification
| Level | ตัวอย่าง | การจัดการ |
|-------|---------|----------|
| **Public** | Product info, pricing | ไม่ต้อง encrypt |
| **Internal** | Employee data, internal docs | Encrypt at rest |
| **Confidential** | User PII, financial data | Encrypt at rest + in transit |
| **Restricted** | Passwords, API keys, secrets | Vault/KMS only, never log |

### Compliance Checklist
* [ ] PDPA compliance (Thailand)
* [ ] Data encryption at rest & in transit
* [ ] Access control & least privilege
* [ ] Audit trail for sensitive operations
* [ ] Data retention & deletion policies

## **MULTI-SERVICE ARCHITECTURE — ระบบหลาย Services**

### Service Communication
| Pattern | ใช้เมื่อ | ตัวอย่าง |
|---------|---------|---------|
| **REST API** | Synchronous, simple CRUD | User service → Order service |
| **Message Queue** | Async, decoupled | Order created → Send email |

### API Versioning
```
/api/v1/users    → version ใน URL (recommended)
/api/users       → Header: Accept-Version: v1
```

### Service Naming
* `user-service`, `order-service`, `payment-service`
* ใช้ kebab-case, ชื่อบอก domain ชัดเจน

### Inter-Service Communication Rules
* **Timeout:** ตั้ง timeout ทุก external call (default: 5s)
* **Retry:** retry with exponential backoff (max 3 times)
* **Circuit Breaker:** ป้องกัน cascading failure
* **Idempotency:** ทุก write operation ต้อง idempotent

### Service Boundaries

## **INCIDENT RESPONSE — การจัดการเหตุฉุกเฉิน**

### Severity Levels
| Level | ผลกระทบ | Response Time | ตัวอย่าง |
|-------|---------|---------------|---------|
| **SEV1** | System down, revenue loss | < 15 min | Payment failed, site down |
| **SEV2** | Major feature broken | < 1 hour | Login broken, checkout stuck |
| **SEV3** | Minor feature broken | < 4 hours | Report not loading |
| **SEV4** | Cosmetic/minor issue | Next business day | UI bug, typo |

### Incident Workflow
```
1. DETECT   → Alert triggered / User report
```
